# Architecture

BuildMart is a **frontend-only single-page app**. There is no server: dummy data is seeded into a
Zustand store and persisted to `localStorage`. Components read from and write to the store; the store
is the only place that "owns" data.

- **Seed data** (`src/data`) is imported once to initialise the store.
- **The store** (`src/store/useStore.ts`) holds all state and every mutation. It is persisted via
  Zustand's `persist` middleware under the key `buildmart-store-v1`.
- **Components** never mutate data directly вЂ” they call store actions, keeping a single source of truth.
- **Selectors** (`src/store/selectors.ts`) provide derived/scoped reads (current user, cart count,
  unread notifications, orders-for-seller, etc.).

## Routing & layouts

Two shells, chosen by route group in `App.tsx`:

- **`MarketplaceShell`** вЂ” buyer experience. Sticky top bar with search, cart, notifications, theme
  & role switcher; **mobile bottom nav** + slide-in menu.
- **`DashboardShell`** вЂ” admin/seller ERP. Collapsible **sidebar** (drawer on mobile), role-aware
  menu (admin sees Customers & Coupons; sellers don't).

Guards:

- `RequireAuth` вЂ” redirects to `/login` when there is no logged-in user.
- `RequireErp` вЂ” additionally requires the active role to be `admin` or `seller`.

## State & persistence notes

- The store is **persisted**, so edits (new orders, stock changes, chats, theme) survive refreshes.
- **Reset** via *Account в†’ Reset data* (calls `resetData()` which re-seeds and clears cart/wishlist).
- IDs for new records are generated by `uid()` in `src/lib/utils.ts` (no crypto dependency).

## Responsiveness

Mobile-first Tailwind throughout. Key patterns:

- Tables on desktop **become cards** on mobile (Products, Customers).
- Sidebar/nav collapse to a **drawer + bottom nav**.
- Charts use Recharts `ResponsiveContainer`.
- The order tracker renders **horizontal** on desktop and a **vertical timeline** on mobile.
